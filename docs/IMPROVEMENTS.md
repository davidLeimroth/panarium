# Panarium improvement plan

Eight work packages, ordered by value over risk. Each package is independently
shippable and has explicit acceptance criteria. Baseline at time of writing:
280 recipes, 70 countries, 1151 built pages, 36 unit tests, Biome clean.

Repo layout reminders:
- Frontend `app/` (Astro 5 + React 19). Data `data/` (recipes, science, schema).
- The recipe schema exists **three times and must stay in sync**:
  `data/schema/recipe.schema.json`, `app/src/lib/recipeSchema.ts` (zod, canonical
  for the build gate), `proto/bread/v1/bread.proto`.
- `app/src/data/` is generated by `scripts/sync-data.mjs` (runs on pre-dev and
  pre-build). Never edit it by hand; never commit it.
- Gates: `cd app && npm test && npm run lint && npm run build`, plus
  `npx tsx scripts/validate-seed.mjs` from the repo root.

## WP1. CI quality gate and licensing

**Problem.** `.github/workflows/deploy.yml` builds and deploys but never runs the
tests, the linter or the seed validator, so a regression ships. The repo is
public with data distilled from cited third-party recipes and has no LICENSE, so
reuse terms are undefined.

**Do.**
1. Add a `check` job to `.github/workflows/deploy.yml` that runs before `build`,
   with `build` gaining `needs: check`:
   - `npm ci` in `app/`
   - `npm run lint` (Biome)
   - `npm test` (vitest)
   - `npx tsx ../scripts/validate-seed.mjs` (run from `app/`, script lives at repo root)
2. Add a `backend` job (matrix not needed): `actions/setup-go@v5` with go 1.24,
   then `go build ./...` and `go vet ./...` in `backend/`. It may run in parallel
   with `check`; `build` does not need to wait on it.
3. Add `LICENSE` at repo root: MIT, copyright "David Leimroth".
4. Add `DATA-LICENSE.md` at repo root explaining the data position in plain
   language: recipe *facts* (quantities, times, temperatures) are not
   copyrightable and are published under CC BY 4.0; method summaries and
   descriptions are original editorial writing under the same terms; every entry
   links its source and the atlas is an index that points outward. Note that
   reusers must keep `source.url` attribution.
5. Link both from `README.md` under a `## License` section.

**Acceptance.** Workflow file contains lint, test and validate steps gating the
deploy; a deliberate failing test would block deployment. `LICENSE` and
`DATA-LICENSE.md` exist and are linked from the README.

**Risk.** Low. No app code changes.

## WP2. Atlas page weight

**Problem.** `app/src/pages/[lang]/breads/index.astro` passes all 280 index
entries as props to the `SearchExplorer` island. Astro server-renders all 280
cards *and* serializes the entire index into the island payload, so the page is
~1.9 MB of HTML, in each of four languages.

**Do.**
1. Emit the search index as a static JSON file per language instead of inlining
   it. Add `app/src/pages/[lang]/breads-index.json.ts` (an Astro static endpoint)
   that `export async function getStaticPaths()` over `LANGS` and returns
   `new Response(JSON.stringify(entries))`. It must reuse `toIndexEntry` from
   `app/src/lib/recipes.ts` so the shape cannot drift.
2. Change `breads/index.astro` to pass only `lang`, `t` and the index **URL**
   (built with `withBase`) to the island. Do not pass `entries`.
3. In `SearchExplorer.tsx`, fetch the index on mount into state; render a
   lightweight skeleton/"loading" state until it arrives. Keep every existing
   behaviour: filters, sort, facet counts, URL state round-trip, result count.
4. Paginate the rendered results: show the first 48 and a "Show more" button that
   appends another 48. Reset to 48 whenever filters or sort change. Add the
   button label to all four dictionaries (`search.showMore`).
5. Keep the island `client:load` (it is the page's whole purpose).

**Acceptance.** `app/dist/en/breads/index.html` is under 250 KB (from ~1900 KB).
`app/dist/en/breads-index.json` exists and parses to 280 entries. In a browser:
the atlas lists breads, filtering by leaven still narrows results and updates
counts, the URL still round-trips filters, and "Show more" reveals further cards.
`npm test` still passes.

**Risk.** Medium, this is the interactive heart of the site. Do not change
`app/src/lib/search.ts` semantics; only change where the data comes from and how
many cards are rendered.

## WP3. Surface formula confidence

**Problem.** 141 of 280 recipes carry `formula.confidence: "estimated"` and the
UI never mentions it. `t.recipe.estimated` is defined in all four dictionaries
but unused.

**Do.**
1. On the recipe page (`app/src/pages/[lang]/breads/[slug].astro`), render a
   small badge next to "The formula" heading when
   `r.formula.confidence === 'estimated'` using the existing `t.recipe.estimated`
   string, and a `t.recipe.fromSource` badge when it is `from-source`. Add
   `recipe.fromSource` to all four dictionaries ("from the source" / "aus der
   Quelle" / "de la fuente" / "d'après la source").
2. Style: reuse the existing `.chip .chip-static` classes, no new visual system.
3. Add a filter: extend `RecipeIndexEntry` in `app/src/lib/search.ts` with
   `sourced: boolean` (true when `confidence === 'from-source'`), set it in
   `toIndexEntry`, add `sourcedOnly: boolean` to `SearchFilters` + `EMPTY_FILTERS`
   + `matches()` + `filtersToParams`/`paramsToFilters` (param key `sourced`).
4. Expose it as a toggle chip in the "Practical" facet group in
   `SearchExplorer.tsx`, labelled by a new `search.sourcedOnly` key in all four
   dictionaries.
5. Add vitest cases in `app/src/lib/__tests__/search.test.ts` covering the new
   filter and its URL round-trip.

**Acceptance.** Recipe pages show a confidence badge. The atlas has a working
"sourced numbers only" toggle that narrows to 139 results and survives a URL
round-trip. Tests pass.

**Risk.** Low.

## WP4. Wire the half-built links

**Problem.** `t.recipe.openInLab` exists in all four dictionaries but is never
rendered, so the recipe to Dough Lab handoff was never built. The fault index,
the School articles and the Lab's warning codes describe the same phenomena but
are not cross-linked.

**Do.**
1. Add an "Open in the Dough Lab" link on the recipe page that carries the
   recipe's formula into the Lab via the query parameters the Lab already parses
   (see the `useEffect` in `app/src/components/react/Calculator.tsx`):
   `st=custom`, `p` (yield.count), `w` (yield.pieceGrams), `h` (hydrationPct),
   `s` (saltPct), `su` (sugarPct or 0), `fa` (fatPct or 0),
   `m` (`levain` when leaven is sourdough, `hybrid` when hybrid, else `yeast`),
   and `blend` as `type:pct|type:pct` from `formula.flours`. Build the href with
   `withBase`.
2. Verify by hand that opening the link reproduces the recipe's hydration and
   salt in the Lab.
3. Cross-link the fault index: in `app/src/lib/scienceData.ts` the `FaultEntry`
   type gains an optional `article?: string` (a School article slug). Populate it
   in `data/science/faults.json` by mapping each fault's `stage` to the most
   relevant existing article slug: mixing -> `flour-and-gluten`, fermentation ->
   `fermentation`, shaping -> `bakers-percentages`, baking ->
   `salt-steam-crust`, storage -> `salt-steam-crust`. Where a fault clearly
   concerns water, use `water-and-hydration`.
4. In the School page fault cards, when `article` is set render a "read more"
   link to `withBase(/{lang}/school/{article}/)`.

**Acceptance.** The recipe page has a Lab link that prefills hydration, salt and
the flour blend. Fault cards link to a School article. Build passes.

**Risk.** Low to medium (the query-parameter contract must match Calculator's
parser exactly).

## WP5. SEO and social

**Problem.** `app/public/` contains only `favicon.svg`. No sitemap, no
robots.txt, no Open Graph or Twitter metadata, so links shared anywhere render
as bare URLs and search engines get no index.

**Do.**
1. Install and enable `@astrojs/sitemap` in `app/astro.config.mjs`. It requires
   `site` to be set, which it already is. Ensure it works with the `base` used
   for the Pages build.
2. Add `app/public/robots.txt` allowing everything and pointing at
   `https://davidleimroth.github.io/panarium/sitemap-index.xml`.
3. Extend `app/src/components/Layout.astro` with Open Graph and Twitter tags:
   `og:title`, `og:description`, `og:type`, `og:url` (canonical absolute URL),
   `og:image`, `og:locale`, `twitter:card=summary_large_image`. Add a `canonical`
   link too. Accept an optional `image` prop defaulting to the site card.
4. Create one static social card at `app/public/og-default.png`. Do not add a
   heavy image pipeline: generate a single 1200x630 PNG. If PNG generation is not
   practical in this environment, ship `app/public/og-default.svg` and reference
   it, noting the limitation in the commit message.

**Acceptance.** `app/dist/sitemap-index.xml` exists and references locale URLs.
`robots.txt` is served. Every page has canonical, og: and twitter: tags with the
correct absolute URL under the `/panarium` base.

**Risk.** Low.

## WP6. Data quality

**Problem.** One true duplicate; batters and doughs are mixed together so derived
statistics are misleading (the flatbread hydration band reported a 1% low end);
some region strings repeat their country.

**Do.**
1. Merge the duplicate: `data/seed/recipes/salt-rising-bread.json` and
   `salt-rising-loaf.json` have identical English names. Keep the better-sourced
   one, delete the other, and fold any distinct detail into the survivor's
   `source.adaptation`.
2. Add `doughType` to the schema in all three places (JSON Schema, zod, proto)
   as an optional enum `dough | batter | paste | stiff`. Semantics:
   - `batter`: pourable, not shapeable (injera, crumpet, farinata, appam,
     roti-jala, johnnycake, mofo-gasy, kisra).
   - `paste`: high-rye or starch pastes that are mixed not kneaded (100% rye
     breads, cassava-bread).
   - `stiff`: very low hydration doughs (bagel, hardtack, candeal, tortilla).
   - `dough`: everything else. Backfill all 280 files. Use hydration plus family
     as the heuristic, then hand-correct the 15 outliers listed by
     `node scripts/derive-formulas.mjs` reasoning (hydration >150 or <30).
3. `scripts/derive-formulas.mjs`: exclude `batter` and `paste` from the
   per-family hydration percentiles and from the fitted relationships, and note
   the exclusion in the emitted JSON as `excluded: {batter: n, paste: n}`.
   Surface nothing new in the UI beyond what already renders.
4. Fix region strings that begin with their own country name (for example
   "Denmark, nationwide" under country DE renders as "Denmark, Denmark,
   nationwide"). Strip the leading duplicate country from `origin.region` across
   the corpus with a script, then delete the script.
5. Re-run `node scripts/derive-formulas.mjs` and commit the regenerated
   `data/science/derived-formulas.json`.

**Acceptance.** `npx tsx scripts/validate-seed.mjs` reports 279/279 valid, 0 hard
failures. Schema is in sync in all three places. The flatbread family band no
longer starts at 1%. No recipe renders a doubled country name.

**Risk.** Medium, it touches every recipe file. Make the backfill a script, run
it, verify, then delete the script so it cannot silently rerun.

## WP7. Localized method steps

**Problem.** Recipe descriptions are translated into all four languages, but
`method[].title`, `method[].summary`, `keeping.note`, `origin.note` and
`ingredients[].note` are English-only. A German visitor gets German chrome and a
German description followed by English instructions.

**Do this as a schema-and-fallback migration, not a translation project.**
1. Schema v2 for `method[]` in all three schema locations: each step becomes
   `{ title: LocalizedText, summary: LocalizedText }` where `LocalizedText`
   requires `en` and allows optional `de`, `es`, `fr`.
2. Migration script: move every existing string into `.en`. Run it over all 280
   files, verify, then delete the script.
3. UI: render `step.title[lang] ?? step.title.en` (same for summary) in
   `app/src/pages/[lang]/breads/[slug].astro`. Add a small note under the method
   list, shown only when the active language is not `en` and the steps have no
   translation for it, using a new dictionary key `recipe.methodEnglishOnly`
   ("Steps are shown in English for now."), translated into de/es/fr.
4. Do **not** attempt to machine-translate 280 x 6 steps in this pass. Leave the
   corpus with `en` populated and the fallback in place, so translations can be
   filled in incrementally without another breaking change.

**Acceptance.** Build passes with the new schema. English pages are unchanged.
German, Spanish and French recipe pages show English steps plus the honest
one-line note. `validate-seed` passes.

**Risk.** High blast radius (every recipe file plus the schema trio), which is
exactly why the translation content itself is deferred.

## WP8. Depth features

Only after WP1 to WP7 are green. Each is independent; implement in this order and
stop when the budget runs out.

1. **Saved formulas.** A "my doughs" shelf in the Lab backed by `localStorage`:
   save the current spec under a name, list saved entries, load, delete. New
   dictionary keys in four languages. No backend.
2. **Pan-volume scaling.** The researched formula already exists in
   `data/science/math-formulas.json`. Add an optional input to the Lab: pan
   length/width/height in cm (or a preset list of common tins), compute pan
   volume, multiply by the sourced dough-density figure, and offer the resulting
   total dough weight as a one-click fill for `pieces x pieceWeight`.
3. **Compare two breads.** A route `/{lang}/compare/?a=slug&b=slug` rendering two
   recipes side by side: crumb portraits, formula table, times, keeping. Add an
   "add to compare" affordance on bread cards. Static page plus a client island.
4. **Bread of the day.** Deterministic pick on the home page from the day number
   so it is stable within a day and needs no server.
5. **Fuzzy search.** Only if the corpus grows past roughly 500 entries: swap the
   substring match in `app/src/lib/search.ts` for MiniSearch. Not needed now.

**Acceptance.** Each feature ships with its dictionary keys in four languages and
does not regress the existing gates.
